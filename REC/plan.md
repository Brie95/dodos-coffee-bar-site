## 2. TAME Risk Analysis  

The TAME framework helps me think through potential risks before they become problems. Here’s what I identified for this project:

---

### **2.1 Technical Risks**  
**Risk:** Limited experience with CSS layout may slow down styling.  
**Mitigation:** Keep the layout simple, use flexbox, and test styling early instead of waiting until the end.

**Risk:** GitHub Pages deployment issues.  
**Mitigation:** Publish the site early in the process so I have time to troubleshoot if needed.

---

### **2.2 Administrative Risks**  
**Risk:** Underestimating the time needed for documentation.  
**Mitigation:** Break documentation into small sections and complete them alongside the website, not at the last minute.

**Risk:** Losing track of files or versions.  
**Mitigation:** Commit changes frequently in GitHub Desktop with clear messages.

---

### **2.3 Management Risks**  
**Risk:** Scope creep (wanting to add extra pages or features).  
**Mitigation:** Stick to the two‑page requirement and save “nice‑to‑have” ideas for later.

**Risk:** Spending too much time on design details early on.  
**Mitigation:** Follow loops: structure → basic styling → content → refinement.

---

### **2.4 External Risks**  
**Risk:** Internet or GitHub downtime near the deadline.  
**Mitigation:** Aim to finish and publish the site at least one day early.

**Risk:** Personal schedule conflicts.  
**Mitigation:** Block out dedicated work time and stick to it.

---

## 3. Work Breakdown Structure (WBS)

### **Level 1 – Project**
1. Plan the project  
2. Design the website  
3. Develop the website  
4. Test and deploy  
5. Reflect and finalize documentation  

---

### **Level 2 – Tasks**

#### **1. Plan the Project**
- 1.1 Review assignment requirements  
- 1.2 Finalize scope statement (`scope.md`)  
- 1.3 Draft project plan (`plan.md`)  

#### **2. Design the Website**
- 2.1 Outline homepage and about page content  
- 2.2 Choose color palette and typography  
- 2.3 Sketch simple layout ideas  

#### **3. Develop the Website**
- 3.1 Build `index.html` structure  
- 3.2 Build `about.html` structure  
- 3.3 Create and link `style.css`  
- 3.4 Add content and adjust layout  
- 3.5 Refine spacing, colors, and readability  

#### **4. Test & Deploy**
- 4.1 Validate HTML and CSS  
- 4.2 Test navigation between pages  
- 4.3 Configure GitHub Pages  
- 4.4 Verify live site and update README  

#### **5. Reflect & Finalize Docs**
- 5.1 Write retrospective (`retrospective.md`)  
- 5.2 Review all documentation for clarity  
- 5.3 Final commit and push

## 4. Schedule Snapshot  
A simple timeline based on typical pacing:

- **Day 1–2:** Planning (scope + plan)  
- **Day 3–4:** Design (layout, colors, content outline)  
- **Day 5–6:** Development (HTML + CSS)  
- **Day 7:** Testing + GitHub Pages deployment  
- **Day 8:** Retrospective + final review  

This schedule keeps the work steady and avoids last‑minute stress.

## 5. Approach Summary  
My approach is intentionally simple: build the structure first, then layer in design, then refine. By working in loops and keeping the scope focused, I can deliver a clean, polished website that reflects the cozy, welcoming vibe of Dodo’s Coffee Bar.
